1. Assignment Overview
Build a full-stack task and project management application called "TaskFlow". The application
should allow users to register and log in, create boards/projects, add tasks, organize them across
status columns (e.g., To Do, In Progress, Done), set priorities and due dates, and track their work
in a clean, responsive interface. Unlike a frontend-only project, you must design and build your
own backend - a REST API, a database, and secure authentication - so that all data persists on
the server and is scoped to the logged-in user.
Think of this as a lightweight Trello / Asana built from scratch. On top of the core CRUD
functionality, you will integrate one small AI feature: a Smart Due-Date / Effort Estimate helper
that uses a free-tier LLM API (called from your backend) to suggest how much effort a task will take
and a reasonable due date, which the user can accept or override.
The goal is to evaluate how you reason across the full stack - data modelling, API design,
authentication, frontend state, and a practical third-party (AI) integration - while shipping something
clean, functional, and deployed.
2. Objectives
This assignment evaluates your skills in the following areas:
- Backend development - designing REST APIs, routing, controllers, and middleware
- Database design - schema/models, relationships, and CRUD operations
- Authentication & security - registration, login, password hashing, JWT/sessions, protected
routes
- Frontend development - React components, hooks, state management, and routing
- Full-stack integration - wiring frontend to backend, handling loading/error states
- Third-party API integration - calling a free-tier LLM API securely from the server
- Code quality - clean, modular, well-organized code across both client and server
- Problem solving & UX - edge cases, validation, and a practical, user-friendly product
3. Tech Stack Requirements

4. Backend, Database & API Requirements
You must build your own backend that exposes a RESTful API and persists data in a database. All
task and board data must be stored server-side and scoped to the authenticated user - do not use
localStorage as the primary data store (it may be used only for the auth token or UI
preferences).
What We Expect
- A clear data model: User, Board/Project, and Task (with relationships)
- Full CRUD REST endpoints for boards and tasks (create, read, update, delete)
- Secure auth: register, login, hashed passwords, JWT issued on login, protected routes via
middleware
- Ownership enforcement - a user can only view/modify their own boards and tasks
- Proper HTTP status codes and consistent JSON responses (including error responses)
- Server-side validation of incoming data (required fields, valid enums for status/priority)
- Clean project structure: separate routes / controllers / models / middleware folders
- Environment variables (.env) for secrets - DB connection string, JWT secret, LLM API key
Suggested Data Model (guidance, adapt as you see fit)
- User: name, email (unique), passwordHash
- Board: title, description, owner (User ref)
- Task: title, description, status (todo / in-progress / done), priority (low / med / high), dueDate,
estimatedEffort, board (ref), owner (ref), timestamps
5. AI Feature - Smart Due-Date / Effort Estimate
Integrate one small AI-powered feature. When a user creates or edits a task, they can click a
"Suggest estimate" button. Your backend sends the task title and description to a free-tier LLM
API and returns a suggested effort estimate (e.g., in hours or T-shirt size S/M/L) and/or a
recommended due date. The suggestion is shown to the user, who can accept it (pre-filling the
fields) or ignore it.
Requirements
- The LLM call must be made from the backend only - the API key must never reach the
browser

Free-tier / Trial LLM APIs (pick any one)
You may use any free-tier or trial LLM API. The choice is yours - document which one you used
and why in your README. Examples of providers with free or trial access:
- Google Gemini API - generous free tier (e.g., gemini-2.0-flash); easiest to start
- Groq - free, fast inference hosting open models (e.g., Llama)
- OpenRouter - single key exposing several free (":free") models
- Anthropic Claude (e.g., claude-haiku-4-5) or Cohere / Mistral - trial credits
Note: Use only free-tier or trial access - do not incur paid charges. If you exhaust a trial, document
it and provide a mock/fallback response so the feature can still be demonstrated.
6. Feature Requirements

- Edit and delete tasks; move a task between columns (button or drag-and-drop)

- Filter/sort tasks by priority or due date; visual cue for overdue tasks
- "Suggest estimate" action that triggers the AI feature (Section 5)
6.4 General / UI Requirements
- Fully responsive design - works well on mobile, tablet, and desktop
- A clean, consistent color palette and cohesive visual theme
